# Charge

import ApiBadge from '@site/src/components/ApiBadge';

**[<ApiBadge type="post" /> /api/interop/v2/charge](https://developer.fiserv.com/product/SnapPay/api/?type=post&path=/api/interop/v2/charge&branch=main&version=V2)**: The Charge API performs a credit card charge or payment using a credit card token to insure the credit card is valid, has the requested credit limit available and to initiate the transfer of funds. This performs the functions of the Authorize API and Capture API in a single call.


:::tip[Tip]
We reviewed the Charge API in the Account Number Tokenizer section. However, this API can be called
independently from the Account Number Tokenizer use case.
:::

To enforce duplicate transaction checks, we are introducing the following account parameter configuration:

1. EnableDuplicateCheck – This configuration ensures the uniqueness of each transaction.
2. DuplicateCheckField – This parameter specifies the field(s) used for the uniqueness check, such as Order ID, Invoice ID, or both.

By adding these configurations, we aim to enhance the integrity of transaction processing by preventing duplicate entries.

|  |  |  |
| --- | --- | --- |
| **accountid**<span className="red-text">*</span> | `number` <br/>  *example: 1000000001* | Pass the SnapPay account ID provided by SnapPay. |
| **userid**<span className="red-text">*</span> | `string` <br/>  *example: user* | User ID sourced from an external application or the SnapPay portal. |
| ecomind | `string` <br/>  *example: E* | Transaction origin indicator for card-not-present transactions. Valid values `T` (telephone/mail), `R` (recurring billing), `E` (e-commerce web/mobile).ecomind at transaction will override what was sent at header level. |
| cof | `string` <br/>  *example: C* | The card on file valid value is either `C` (Customer Initiated Transaction) or `M` (Merchant Initiated Transaction). For Card Not Present transactions the value can be either `C` or `M` at header level and it will be copied to all transaction(s) within payload. For Card Present the value needs to be sent at transaction level (instead of header) and it should be `C` for first transaction and M for subsequent transaction. COF at transaction will override what was sent at header level. |
| cofscheduled | `string` <br/>  *example: N* | Indicates if card on file is scheduled. Valid values `Y` (consent given), `N` (no consent). cofscheduled at transaction will override what was sent at header level. |
| **transactions**<span className="red-text">*</span> | `array[object]` | This section contains an array of transaction details, which can include single or multiple transactions. |

</TabItem>

| message | `string` <br/>  *example: Transaction successful.* | Provides a detailed status message for the transaction, offering additional context or information about the transaction outcome. |
